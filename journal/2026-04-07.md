# 2026-04-07

## Editor instrumentation for activity tracking

Wired Emacs into a local-first activity tracking pipeline (Wakapi for
coding heartbeats, ActivityWatch for window/idle, keyfreq for command
frequency, org-clock for task time) and disabled the PostHog telemetry
module.

### Added to `init.el`

A new "Activity tracking" section with five `use-package` forms:

- **`wakatime-mode`** — heartbeats to a self-hosted Wakapi instance.
  Guarded with `:if (executable-find "wakatime-cli")` so the config
  still loads cleanly on hosts without the CLI. Configuration of
  `~/.wakatime.cfg` (api_url, api_key) is a host-side concern, not
  jotain's.
- **`activity-watch-mode`** — heartbeats to ActivityWatch. Activated
  unconditionally; the package degrades gracefully when the
  `aw-server` isn't reachable, so it's safe even on hosts that don't
  run the daemon.
- **`keyfreq`** — command frequency tracking, with the persistent
  autosave mode enabled so stats survive restarts. Data lands in
  `~/.emacs.keyfreq` and is consumable by external analysis tools.
- **`org-clock`** — built-in, but now wired with persistence
  (`org-clock-persist`, `org-clock-idle-time 15`,
  `org-clock-into-drawer`) and `org-clock-persistence-insinuate`.
  Loaded eagerly (no `:defer t`) so the persistence hooks attach at
  startup — otherwise a clock that was running in the previous
  session won't be restored to the mode line until something else
  triggers org-clock. First pass had `:defer t` here and got this
  wrong.
- **`org-clock-csv`** — exporter, deferred via `:defer t` so the
  package isn't loaded eagerly but its commands (e.g.
  `org-clock-csv-export`) remain autoloaded and discoverable from
  `M-x`. First pass used `:after org-clock`, which blocks autoload
  registration until org-clock is loaded and hides the commands from
  `M-x` discovery; `:defer t` achieves the same "don't load eagerly"
  goal while preserving autoloads.

### Disabled `jotain-telemetry`

`jotain-telemetry-mode` (PostHog batch upload of command counts, mode
times, session events, errors) is now turned off via
`(jotain-telemetry-enabled nil)` and the explicit `(jotain-telemetry-mode 1)`
call removed. The module file stays in `lisp/` for now — only the
init-time activation is gone. The new tracking stack covers the same
ground locally, without shipping events to a third party.

### Design choices

- **`:custom` over `setq`**: enforced by the `elisp-conventions`
  skill — `setopt`/`:custom` runs the defcustom `:set` function and
  validates `:type`, neither of which `setq` does. So org-clock
  customizations go through `:custom`.
- **Conditional `wakatime-mode`**: activating `global-wakatime-mode`
  unconditionally throws errors at startup on machines without
  `wakatime-cli`. `executable-find` is the feature-gate; this matches
  jotain's preference for `fboundp`/runtime detection over hard
  version checks.
- **No new `lisp/jotain-tracking.el` module**: the wiring is purely
  declarative `use-package` forms with no custom logic, so it lives in
  `init.el` alongside `nix-ts-mode`, `markdown-mode`, and `magit`.
  `jotain-telemetry.el` exists as its own module because it implements
  custom event-collection logic; the activity-tracking packages are
  off-the-shelf, so a wrapper module would be ceremony with no payoff.

## Font configuration

Added a font preference list with automatic fallback detection. On startup
(and when creating new `emacsclient` frames), Emacs walks the list and
picks the first font that's installed:

1. JetBrainsMono Nerd Font (11pt)
2. FiraCode Nerd Font (11pt)
3. Iosevka Nerd Font (11pt)
4. CaskaydiaCove Nerd Font (11pt) — Nerd Fonts' patched name for Cascadia Code
5. Hack Nerd Font (11pt)
6. DejaVu Sans Mono (10pt) — final fallback, widely available

All six fonts are now declared in `devenv.nix` `packages`, so `devenv shell`
makes them available without polluting the system font set.

The `server-after-make-frame-hook` ensures the font is applied correctly
when connecting via `emacsclient`, since `find-font` needs a live frame.

## Ghostty terminal compatibility

Made `emacs -nw` work properly inside Ghostty (and other modern terminals
like Kitty/WezTerm/Alacritty as a side benefit). Four changes across
`early-init.el` and `init.el`.

### `term-file-aliases` for `xterm-ghostty` (early-init.el)

Ghostty advertises `TERM=xterm-ghostty`. Emacs looks up `term/$TERM.el`
during `tty-run-terminal-initialization' to enable per-terminal extensions
(modifyOtherKeys, bracketed paste, focus-in/out events, 24-bit color
detection). Without a matching file Emacs falls back to a minimal setup
and silently misses all of those.

Aliasing `xterm-ghostty` to `xterm-256color` makes `term/xterm.el` load
instead, which is the right base for Ghostty (it's an xterm-compatible
terminal that also speaks COLORTERM=truecolor).

This *had* to go in `early-init.el`, not `init.el`: terminal initialization
runs before init.el is loaded for `emacs -nw`, so an alias added in init.el
would arrive too late for the initial frame.

Long-term, Emacs master has merged `term/xterm-ghostty.el` upstream, so
once jotain moves to a build that includes it the alias becomes
redundant. Cheap enough to leave in place as a safety net.

### Mouse, KKP, OSC 52 clipboard (init.el)

New `;;;; Terminal compatibility` section between the `emacs` use-package
block and `nix-ts-mode`:

- **`xterm-mouse-mode`** via a `tty-setup-hook` (named function
  `jotain-tty-setup`, not a lambda — `elisp-conventions` skill). Using the
  hook instead of a one-shot `(unless (display-graphic-p) ...)` means new
  `emacsclient -t` frames against a GUI daemon also get mouse support,
  not just the initial frame.

- **`kkp`** (Kitty Keyboard Protocol) — lets Emacs finally distinguish
  `C-i`/`TAB`, `C-m`/`RET`, `C-[`/`ESC`, shifted function keys, and
  modified arrow keys in the terminal. Ghostty implements the protocol.
  `global-kkp-mode` is a no-op on GUI frames, so it's enabled
  unconditionally and handles mixed GUI/tty daemon setups correctly.

- **`clipetty`** — OSC 52 clipboard sync. Yank/kill cross the
  terminal-host boundary, including through SSH and tmux. Wired via
  `:hook (after-init . global-clipetty-mode)`.

### Design choices

- **`:ensure t` for `kkp` and `clipetty`**: matches the existing pattern
  in `init.el` (consult, magit, markdown-mode etc. are all installed via
  package.el from MELPA, not via Nix). The `jotain-architecture` skill
  describes a Nix-managed end state, but the current init.el is package.el
  + MELPA, so following the in-file convention rather than the
  aspirational one.
- **No `:if (not (display-graphic-p))` guards**: both modes already
  handle GUI/tty correctly internally, and the use-package `:if` only
  evaluates once at config time, which is wrong for daemon setups. Better
  to load and let the modes self-gate.
- **Section placement**: terminal-compat sits right after the core
  `emacs` use-package block because it's foundational config that
  everything else can assume is in place.

### What this *doesn't* fix

- **Undercurl / styled underlines**: Ghostty draws them, but Emacs's tty
  face renderer doesn't emit the SGR codes for curly/dotted/dashed
  underlines, so they show as plain underlines in `emacs -nw`. GUI only.
- **Ligatures**: handled by Ghostty's font renderer regardless, no
  Emacs-side change needed.
- **Font preferences**: `jotain-set-preferred-font` is a no-op in tty
  frames (`find-font` returns nil for everything). That's correct
  behavior, just worth knowing.

## Fix startup failure: `void-variable package-archives`

Emacs was failing at startup with:

```
Debugger entered--Lisp error: (void-variable package-archives)
  (member '("melpa" . "https://melpa.org/packages/") package-archives)
```

The call `(add-to-list 'package-archives ...)` in `init.el` was executed before `package.el` was loaded. With `lexical-binding: t`, `add-to-list` expands to an inline `member` check that directly evaluates `package-archives` — if the variable hasn't been defined yet, that's a `void-variable` error.

The fix: add `(require 'package)` immediately before the `add-to-list` call. This ensures `package.el` is loaded and `package-archives` is defined with its default value before we append MELPA to it.
