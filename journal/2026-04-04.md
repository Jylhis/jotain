### The problem (from the skills plan)

Claude has five catastrophic failure modes when working with Emacs and Nix-managed configs:

1. **Package hallucination** — confidently suggests packages that don't exist in nixpkgs or have different attribute names. In a Nix-managed Emacs, `M-x package-install` isn't an option, so a wrong name means a build failure and a wasted `nix build` cycle.

2. **Non-idiomatic Elisp** — missing lexical-binding headers, `setq` where `defcustom` belongs, wrong hook conventions (lambdas instead of named functions), misuse of `eval-after-load` vs `with-eval-after-load`, `require` where autoloads suffice. Code works but is structurally wrong.

3. **Nix-Emacs impedance mismatch** — doesn't internalize that package management is Nix's job. Suggests `straight.el` or `package.el` patterns where the overlay handles everything. Doesn't understand emacs-overlay, tree-sitter grammar packaging, or native-comp build flags.

4. **Architecture blindness** — without knowledge of jotain's module structure, generates flat config blobs instead of properly wired modules. Puts everything in one file instead of following the modular layout.

5. **Generic debugging advice** — "Check your init.el" is useless when the config is a Nix flake. The debugging workflow for Nix-managed Emacs is fundamentally different: bisect modules, check Nix evaluation, inspect the store path, trace through overlay composition.

### Research phase: methodology

Created a deep research prompt (`emacs-deep-research-prompt.md`) that establishes a methodology: **Triangulate → Trace → Synthesize → Verify** using three instruments:

1. **Live Emacs** (`emacs --batch --eval '...' 2>/dev/null`) — runtime ground truth. Each invocation is a fresh process. Use `princ` for stdout, temp files for large output.
2. **Emacs source code** (C core in `src/`, Elisp stdlib in `lisp/`, NEWS files in `etc/`) — implementation details, call chains, undocumented behavior.
3. **HTML documentation** (Elisp Reference Manual: 1086 nodes, GNU Emacs Manual: 793 nodes) — conceptual framework, intended usage, cross-references.

The methodology explicitly says: never answer from a single source. Documentation can be outdated, source code lacks context, and runtime behavior may differ from both due to advice, hooks, or configuration. When sources disagree, note the discrepancy.

### Research phase: key discoveries

**Elisp conventions:**
- Unused variable detection ONLY works with `lexical-binding: t` — byte-compiler can't track variable usage under dynamic binding. This makes the lint hook's lexical-binding check severity Error, not Warning.
- `thread-first`/`thread-last` still need `(require 'subr-x)` in Emacs 30 while `when-let*`/`if-let*`/`and-let*` moved to core — inconsistency that's easy to get wrong.
- `setopt` calls the `:set` function AND validates `:type`. `setq` does neither. So `(setq custom-var value)` may silently skip important side-effects the defcustom author intended.
- `if-let`/`when-let` (without star) are obsoleted in Emacs 31. The starred versions (`if-let*`/`when-let*`) are canonical.
- `seq-*` functions are available without require. `map-*` still needs `(require 'map)`.
- `cl.el` is in the `obsolete/` directory. Will be removed entirely. Use `cl-lib`.
- `add-hook` DEPTH parameter (Emacs 28+) replaced the old APPEND boolean. Non-nil symbol = depth 90 for backward compatibility.
- `defcustom` vars are ALWAYS dynamically bound, even with `lexical-binding: t` (they use `defvar` internally).
- Byte-compilation catches: undefined functions, wrong arg counts, unused lexical vars, free variable refs, interactive-only calls, obsolete API usage. Categories in `byte-compile-warnings`: `callargs`, `constants`, `docstrings`, `free-vars`, `ignored-return-value`, `interactive-only`, `lexical`, `obsolete`, `unresolved`, etc.
- `checkdoc-file` checks documentation string quality. `package-lint` is NOT built-in (external, MELPA).

**Keybinding system:**
- The full 8-layer keymap lookup order: (1) `overriding-terminal-local-map`, (2) `overriding-local-map`, (3) `keymap` text/overlay property at point, (4) `emulation-mode-map-alists` (Evil mode lives here), (5) `minor-mode-overriding-map-alist`, (6) `minor-mode-map-alist`, (7) local keymap (major mode), (8) global keymap.
- After lookup, command remapping applies: `[remap original-cmd]` → `new-cmd` entries in a keymap. Only one level of remapping.
- `C-c <letter>` (a–z, A–Z) are the ONLY sequences reserved exclusively for users. Verified: all 26 lowercase keys are free in vanilla Emacs.
- `keymap-set` uses `key-valid-p` strings (human-readable like `"C-c a"`). `define-key` uses internal format (vectors, old escape syntax). Modifier order must be alphabetical: `A-C-H-M-S-s`.
- `defvar-keymap` (Emacs 29+) supports `:repeat t` for `repeat-mode` integration.
- `minor-mode-key-binding` returns an alist of `(modename . binding)` — key for conflict detection.
- `accessible-keymaps` returns all reachable sub-keymaps from a prefix — useful for enumeration.

**ERT testing:**
- ERT deliberately omits JUnit-style fixtures. Uses `let` for binding, `unwind-protect` for cleanup, custom macros for composable fixtures.
- Key utilities from `ert-x.el`: `ert-with-temp-file`, `ert-with-temp-directory`, `ert-with-message-capture`, `ert-simulate-command`, `ert-simulate-keys`, `ert-run-idle-timers`.
- Function mocking via `cl-letf`: `(cl-letf (((symbol-function 'fn) (lambda ...))) body)`.
- Batch exit codes: 0 = pass, 1 = failure, 2 = internal error.
- Buttercup available as `emacsPackages.buttercup` but recommendation is to stick with ERT — built-in, no extra deps, jotain already uses it.

**Debugging & profiling:**
- `--debug-init` sets `init-file-debug` to `t` and binds `debug-on-error` to `t` ONLY during init file loading, bypassing the normal `condition-case`. In batch mode, errors print backtraces to stderr and exit with code 255.
- `SIGUSR2` via `debug-on-event` enters the debugger even when `inhibit-quit` is set — the primary mechanism for debugging a frozen Emacs.
- Three profiling systems: sampling profiler (`profiler-start 'cpu`/`'mem`/`'cpu+mem`), ELP (deterministic per-function timing), and `benchmark-run` (micro-benchmarks).
- `handler-bind` (Emacs 30+) runs handlers in the error's dynamic context, allowing backtrace capture. `condition-case` unwinds the stack first.
- Native compilation: 14 key variables including `native-comp-debug` (0–3), `native-comp-verbose` (0–3), `comp-abi-hash`. Recompilation storms diagnosed by checking eln-cache writability and ABI hash mismatches.
- Default `gc-cons-threshold` is 800000 (~800KB). Batch mode raises `gc-cons-percentage` from 0.1 to 1.0. `memory-report` available in Emacs 29+.

**gptel & mcp.el:**
- Emacs has NO sandboxing for `eval`. Any evaluated code has full access to: filesystem, network, process execution, all Emacs state. Self-improving Emacs must use Nix as validation gate, `:confirm t` on tools, git for rollback.
- gptel's `gptel-make-tool` supports `:confirm t` (ask before executing), `:include t` (include result in context), `:async` (non-blocking execution).
- mcp.el uses `jsonrpc-process-connection` (built-in `jsonrpc.el`) for stdio transport.
- `auth-source-search` reads `.authinfo.gpg` for secure API key storage.

**Nix-Emacs packaging:**
- `emacsPackagesFor` takes an Emacs derivation → package scope. Merging priority: manual > GNU ELPA > NonGNU ELPA > MELPA Stable > MELPA > GNU-devel ELPA.
- `withPackages` generates `site-start.el` (sets `load-path`, `treesit-extra-load-path`, runs autoloads), wraps the binary via `makeWrapper`.
- Jotain's `needsOverride` guard preserves Hydra binary cache for plain mainline builds — building from source takes 10+ minutes.
- Built-ins to NEVER add as packages: `use-package`, `eglot`, `which-key`, `modus-themes`, `project`, `flymake`, `treesit`, `seq`, `map`, `xref`, `jsonrpc`, `so-long`.
- `mu4e` is part of `pkgs.mu`, NOT `emacsPackages`. `notmuch` is part of `pkgs.notmuch`.

**NEWS archaeology:**
- 15 NEWS files back to Emacs 1 (1986). Format: `*` headings (outline-mode compatible), `**` entries, `+++`/`---` manual-update markers.
- Feature-gating with `fboundp` is more robust than version checks because it handles custom builds where features may be compiled out.

### Gap analysis: plan vs reality

The skills plan (`emacs-skills-plan.md`) was written before the jotain project structure was fully analyzed. It assumed:
- A Nix module system with `config.jotain.modules.X.enable` and `lib.mkIf cfg.enable` patterns
- Multiple modules organized by concern (completion, git, lsp, ui, etc.)
- The `jotain-architecture` skill describing a rich modular Nix structure

**Reality (from the jotain structure agent):**
- Flat structure: `lisp/` for custom modules, `test/` for ERT tests
- Only ONE custom module exists: `jotain-telemetry.el`
- Packages managed via `elpa/` (not fully Nix-managed yet): nix-mode, magit-section, llama, cond-let
- `use-package` in `init.el` as the primary configuration mechanism
- `default.nix` wraps `emacs.nix` with 275 tree-sitter grammars but no Elisp packages from Nix yet

**Decision:** All agents and skills were designed for the CURRENT flat structure, not the aspirational modular Nix one. The module scaffolder creates files in `lisp/` and `test/` and wires `use-package` in `init.el` — not Nix modules.

### Design decisions

1. **Agents vs skills boundary** — agents need tool access (emacs --batch, grep, read); skills encode static knowledge/rules. If it needs to run commands to answer, it's an agent.

2. **Two-phase lint** — Phase 1 is regex (< 100ms, no Emacs needed), Phase 2 is byte-compile (1–5s). The hook script only does Phase 1 for speed on every edit. The elisp-reviewer agent does both phases for thorough review.

3. **`C-c j` as jotain prefix** — the keybinding-analyzer establishes this. `C-c <letter>` is the only user-reserved range in Emacs. Subkeys: `C-c j t` for telemetry, `C-c j g` for git, etc.

4. **Flat file structure, not Nix modules** — the aspirational skills plan assumed modular Nix with `config.jotain.modules.X.enable`. Current reality is `lisp/` + `test/` + `use-package`. Built for reality.

5. **`setopt` over `setq` for defcustoms** — research confirmed `setopt` calls the `:set` function and validates `:type`. `setq` does neither. Enforced by conventions skill, checked by lint hook.

6. **Feature-gating over version-gating** — `(fboundp 'keymap-set)` is more robust than `(>= emacs-major-version 29)` because it handles custom builds where features may be compiled out.

7. **No sandboxing for self-improving Emacs** — Emacs has zero eval sandboxing. Mitigations documented in gptel-mcp-workflow: Nix as validation gate (changes must survive `nix build`), `:confirm t` on tools, git as rollback.

8. **Opus model for researcher agent** — research tasks need the most capable model for synthesis across three sources and deep reasoning about Emacs internals.

9. **Research prompt as reusable agent** — the 467-line deep research prompt was the most valuable artifact. Instead of one-off use, it became the `emacs-researcher` agent, permanently available.

10. **Skill descriptions are "pushy"** — Anthropic's own documentation warns that Claude tends to under-trigger skills. Descriptions explicitly list trigger phrases and say things like "If you are about to write even a single line of Elisp, consult this skill first."
