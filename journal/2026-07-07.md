## Integrate origin's devenv/terminal work with local Go/eglot work (best-of-both merge)

`origin/main` and the local branch had both diverged from `10922cb` and
independently reworked the Go tooling plus the shared `init-prog` /
`init-project` / `init-keys` files. Rather than discard either side, the
two were merged and the stronger implementation kept on each overlap.

### What origin brought

- `lisp/devenv.el` + `lisp/init-devenv.el` — native devenv.sh integration
  (transient on `C-c v`: tasks, scripts, processes dashboard, test/build
  through compilation-mode, env reload, `devenv lsp` for devenv.nix,
  `devenv mcp` for gptel, auto-activation trust + mode-line status).
- `lisp/init-terminal.el` — the ghostel terminal emulator (libghostty-vt),
  replacing vterm, plus the kkp/clipetty/tty-tip block moved out of
  `init-ui.el`.
- dape with `C-x C-a` (gud prefix), right-side windows, and breakpoint
  persistence in `var/dape-breakpoints`.

### What local kept

- `jotain-prog--maybe-eglot-ensure`: eglot auto-starts for *any* project
  file whose language server is on the buffer's (envrc-applied) PATH, so
  enabling a language in a project's devenv lights up its LSP with no
  per-language config. Function-valued `rass` resolvers run discovery at
  connect time in the project env instead of once at startup.
- The classic-vs-tree-sitter mode diagnostic (`jotain-prog--warn-non-ts-mode`).
- `envrc` command map on `C-c e`.
- `go-tag` struct-tag editing alongside `gotest`.

### Overlap resolutions

- **Go modes**: origin's file structure (buffer-local gopls workspace hook,
  go.work fallback, `go-ts-mode-indent-offset 8`) with the local branch's
  richer gopls settings (`completeUnimported`, `unusedparams`/`shadow`/
  `nilness`/`unusedwrite` analyses).
- **Formatter**: `goimports` (origin) over `gofmt` — adds import management.
- **dape**: origin's `C-x C-a` + persistence over the local `C-c d` prefix.
  Because docker.el was removed on the local branch, dirvish keeps its
  long-standing `C-c d` with no clash, so there was no need to move it.
- **compile-multi**: union of both sides' Go commands
  (`test`/`test current`/`test -race`/`build ./...`/`run`/`vet`/`golangci-lint`)
  plus origin's `go-mod-ts-mode` (`mod tidy`/`mod download`) and the
  `devenv test`/`build`/`up` entries, and `devenv.nix` as a project root

## Fix: "File mode specification error: Cannot open load file … go-mode"

### Symptom

Opening Emacs (built from the remote branch) and visiting a `.go` file
produced:

```
File mode specification error: (file-missing Cannot open load file No such file or directory go-mode)
```

### Root cause

Both branches deliberately dropped the classic `go-mode` package in favour
of the built-in `go-ts-mode`, so nothing installs `go-mode` directly (it
only ever arrives as a transitive dependency of `gotest`/`go-tag`). But a
`.go` file can still be routed to the classic `go-mode`:

- a **stale `var/package-quickstart.el`** carrying `go-mode`'s
  `;;;###autoload (add-to-list 'auto-mode-alist '("\\.go\\'" . go-mode))`
  after the elpa copy of `go-mode` was cleaned, or
- a **treesit-auto grammar-missing fallback**, or
- an old `auto-mode-alist` entry.

`set-auto-mode` then calls `go-mode`, which `require`s a package that is not
on `load-path`, and the whole file-visit aborts with the mode-spec error.

### Fix

Add `major-mode-remap-alist` entries in `init-lang-go.el` routing classic
`go-mode`/`go-mod-mode` to their tree-sitter equivalents:

```elisp
(when (fboundp 'go-ts-mode)
  (dolist (remap '((go-mode         . go-ts-mode)
                   (go-dot-mod-mode . go-mod-ts-mode)
                   (go-mod-mode     . go-mod-ts-mode)))
    (add-to-list 'major-mode-remap-alist remap)))
```

`set-auto-mode` consults `major-mode-remap-alist` *before* it loads the
mode function, so `go-mode` is never required and every Go buffer lands in
`go-ts-mode` regardless of how the file got routed. In the full
distribution the Go grammar is always present, so the remap resolves to a
working `go-ts-mode`; if the grammar were ever missing the failure would be
a clear treesit-grammar error rather than a load error that breaks all
file-visiting.

## Fix: custom.el supersession prompt

### Symptom

```
custom.el changed on disk; really edit the buffer? (y, n, r or C-h) n
userlock--ask-user-about-supersession-threat: File changed on disk: ~/.config/emacs/var/custom.el
```

### Root cause

`custom-file` is write-only (`init.el`): the declarative config in git is
the single source of truth, and `custom.el` is regenerated from scratch on
every `custom-save-all` — which `package.el` triggers whenever it persists
`package-selected-packages` after installing a package. When more than one
session writes it (a daemon and its `emacsclient` frames, or two Emacsen
sharing `var/`), the file changes on disk underneath the writer.
`custom-save-variables` then modifies its buffer, Emacs notices the visited
file changed on disk, and raises the supersession prompt (aborting the
write with a `file-supersession` error if declined).

### Fix

Because the file is disposable, clobbering it is the correct outcome.
`init-core.el` wraps `custom-save-all` so the supersession asker is a no-op
for the duration of the save:

```elisp
(defun jotain-core--custom-save-without-supersession (orig &rest args)
  (let ((saved (symbol-function 'ask-user-about-supersession-threat)))
    (unwind-protect
        (progn
          (fset 'ask-user-about-supersession-threat #'ignore)
          (apply orig args))
      (fset 'ask-user-about-supersession-threat saved))))
(advice-add 'custom-save-all :around
            #'jotain-core--custom-save-without-supersession)
```

## Fix: go-mode crash recurs on *every* file; switch env from direnv to native devenv; tree-sitter cleanup

Three problems surfaced running `just run-built` and opening files. The
first *completes* the "Cannot open load file … go-mode" entry above, whose
`major-mode-remap-alist` fix turned out to be insufficient.

### 1. `Cannot open load file … go-mode` on a **Nix** file (not just Go)

The earlier remap-only fix did not hold: opening a `.nix` file still raised
the error and nix-mode never loaded. Real root cause: `go-tag`/`gotest`
drag in classic `go-mode`, whose autoloads install a **global**
`magic-mode-alist` entry `(go--is-go-asm . go-asm-mode)`. `magic-mode-alist`
is consulted for **every** file `set-auto-mode` visits, and `go--is-go-asm`
is autoloaded from `go-mode`, so opening *any* file force-loads classic
go-mode. When go-mode isn't on `load-path`, the load fails and mode setup
aborts. `major-mode-remap-alist` does **not** prevent this — it only
rewrites the *chosen* mode; the magic predicate fires earlier and loads
go-mode directly.

Fix (`init-lang-go.el`): strip go-mode's global autoload state at module
load — remove the `go--is-go-asm` `magic-mode-alist` entry and the classic
`.go`/`go.mod`/`go.work` `auto-mode-alist` entries (the module already
routes those to the tree-sitter modes). The remap stays as a backstop.
Verified in the built Emacs: `flake.nix` → `go--is-go-asm` gone, opens in
`nix-ts-mode`; `.go` still opens in `go-ts-mode`.

### 2. gopls `go: executable file not found in $PATH` → disable direnv, use native devenv

`Direnv failed in .../otherproject/` meant envrc applied no env, so gopls (global)
couldn't find `go`. Trigger: `just run-built` launches Emacs *inside*
jotain's own devenv shell (leaking `DEVENV_*`/`NIX_*`), so envrc's nested
`use devenv` for another project fails. Decision: stop using direnv/envrc;
load the environment natively via devenv.

- `devenv.el`: new `devenv-env-defer-to-direnv` defcustom (default `t`,
  keeps the library reusable); `devenv-env--turn-on` only skips
  `.envrc`/`envrc-mode` buffers when it is non-nil. New `devenv-env-loading-p`
  predicate reports an in-flight `print-dev-env` fetch.
- `init-prog.el`: removed the `envrc` `use-package` block; removed the
  curated per-mode `:hook . eglot-ensure` (it fired *before* the globalized
  `devenv-env-mode` turned on, so eglot connected with the global env).
  Hardened `jotain-prog--maybe-eglot-ensure` to call `eglot-ensure`
  unconditionally while the devenv env is loading (the loader's deferral
  advice holds and replays it once the env lands), keeping the
  `executable-find` guard only for the settled path.
- `init-devenv.el`: enable `devenv-env-global-mode` on `after-init`, set
  `devenv-env-defer-to-direnv` nil.

Trust: `devenv-env-mode` only activates for projects trusted with
`devenv-allow` (`C-c v`); until then the mode line shows `devenv[!]`.
Trade-off: pure-direnv, non-devenv projects lose auto-env. Verified: the
loader enables cleanly and installs the `eglot-ensure` deferral advice.
(Interactive confirmation that gopls finds `go` in otherproject, and whether
otherproject is a devenv project, is the remaining user step.)

### 3. Tree-sitter cleanup

The `TREE_SITTER_DIR` → `treesit-extra-load-path` block in `early-init.el`
was dead: nothing set that env var. Grammars come from Nixpkgs' own
`site-start.el` (`treesit-extra-load-path` → `…/emacs-packages-deps/lib/`).
Removed the block, fixed the stale comment in `init-prog.el`, and stopped
the async native-comp "grammar for go unavailable" warning by propagating
the path into workers (which run without site files):

```elisp
(setq native-comp-async-env-modifier-form
      `(setq treesit-extra-load-path ',treesit-extra-load-path))
```

### Note: stale `lisp/*.elc` shadow source edits

`early-init.el` sets `load-prefer-newer` to `noninteractive`, so an
*interactive* Emacs prefers `.elc` even when the `.el` is newer. A prior
`just compile` had left stale `lisp/*.elc` (devenv, init-devenv,
init-lang-go, …) that would shadow these edits. Ran `just clean` to clear
them and the eln-cache; the next `just run-built` recompiles from source.
`nix flake check`'s elisp-compile (warnings-as-errors) passes with the edits.
