# 2026-07-23

## Fix `Jinx: No dictionaries available for "en_US"` — bundle aspell dicts

### Summary

Resolved the long-standing jinx spell-check error, deferred on
[2026-06-09](2026-06-09.md) and [2026-07-20](2026-07-20.md) as a
"deployment gap." The distribution now bundles the aspell dictionaries it
needs, so `just run-built`, `just screenshot`, and every module consumer
spell-check out of the box.

### Corrected root cause

The earlier notes were wrong on two counts, verified against the pinned
nixpkgs:

- **enchant *does* have the aspell backend.** `enchant-2.6.9` is built with
  `--with-aspell` (aspell sits in `propagatedBuildInputs`), alongside
  hunspell/nuspell/voikko. There was never a missing-backend problem.
- **The distribution shipped no dictionary data.** `jotainEmacsPackages*`
  (`nix/mk-overlay.nix`) pulled in the enchant + aspell *libraries* via the
  `jinx` epkg, but nothing added an `aspell-dict-*` package to the closure.
  jinx's C module therefore loaded, enchant initialised, and libaspell found
  no `en_US` dictionary anywhere → the error.

The modules install `pkgs.aspellDicts.en` into a profile and rely on
nixpkgs' libaspell `data-dirs-from-nix-profiles.patch` to find
`$NIX_PROFILES/*/lib/aspell` at runtime. That works only when
`NIX_PROFILES` is populated at launch — so a bare `./result/bin/emacs`
(the `just run-built` path) or a systemd/launchd daemon with an empty
`NIX_PROFILES` still failed.

### Why `NIX_PROFILES` is not enough (the trap the modules fell into)

nixpkgs' libaspell `data-dirs-from-nix-profiles.patch` feeds
`$NIX_PROFILES/*/lib/aspell` into libaspell's data-dir list *only* for
dictionary **enumeration**. So `enchant_broker_list_dicts` (and
`enchant-lsmod-2 -list-dicts`) happily report `en_GB (aspell)` when an
`aspellDicts.en` is on `NIX_PROFILES` — but `enchant_broker_request_dict`,
the call jinx actually makes, builds a *speller*, whose master word list
resolves under the default `data-dir` (`${aspell}/lib/aspell/en`, which does
not exist). Result: enumeration passes, the speller fails, jinx still shows
"No dictionaries available." Verified directly:
`enchant-2 -d en_GB -l` with only `NIX_PROFILES` set →
`The file "…/aspell-0.60.8.2/lib/aspell/en" can not be opened for reading`.
This means the modules' profile-installed `aspellDicts.en` never actually
worked for spell-checking either — only for the dict list.

### Changes

- **`nix/mk-overlay.nix`** — build one `aspellWithDicts (d: [en fi de fr])`
  directory (libaspell data files + all four dicts under a single
  `lib/aspell`) and, in the `mkJotainEmacsPackages` re-wrap loop, set
  `ASPELL_CONF="dict-dir <env>/lib/aspell; data-dir <env>/lib/aspell"` via
  `makeBinaryWrapper --set-default`. Pointing both `dict-dir` and `data-dir`
  at the combined directory is what makes `request_dict` succeed. Verified:
  `enchant-2 -d en_GB -l` accepts "colour", flags "helllo"; and inside the
  wrapped Emacs `jinx--mod-dict` returns non-nil for en_GB/fi/de/fr.
  `--set-default` lets a user (or a module wrapper) still override
  `ASPELL_CONF` to add languages.
- **`lisp/init-writing.el`** — set `jinx-languages` to `"en_GB"` (British
  English) as the default. fi/de/fr are bundled and reachable per buffer via
  `C-M-$`; a single default avoids foreign words silently passing as correct
  English.

### Notes

- One edit covers all three distribution builds (`jotainEmacsPackages`,
  `jotainEmacsPackagesNoGui`, `jylhisEmacsPackages`) — they share the
  re-wrap loop. Module consumers (`module.nix`, `module-system.nix`,
  `module-nix-on-droid.nix`) exec this same wrapped binary without setting
  `ASPELL_CONF`, so the `--set-default` applies and they get working
  spell-check for the four languages with no module change.
- Cache parity is untouched: the change lives in the custom
  `jotainEmacsPackages` derivation, not `emacs.nix`, so the bare-Emacs
  store-path parity invariant is unaffected.
- **Follow-up:** the modules' `services.jotain.spell.dictionaries` option
  and the hardcoded `aspellDicts.en` in `module-system.nix` /
  `module-nix-on-droid.nix` (and the "libaspell finds `$profile/lib/aspell`"
  comments) are now known to be no-ops for actual spell-checking. To make
  extra languages configurable, a later change should build an
  `aspellWithDicts` from that option and export `ASPELL_CONF` in the module
  wrappers, replacing the profile install.
