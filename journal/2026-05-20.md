# 2026-05-20

## `jotain.service` aborts with `vc-git-root` void

### Symptom

The `jotain` systemd user service exited with `status=255/EXCEPTION` shortly
after `init.el` finished loading:

```
Saving file /home/user/.config/emacs/var/custom.el...
Wrote /home/user/.config/emacs/var/custom.el
Symbol's function definition is void: vc-git-root
Wrote /home/user/.config/emacs/var/recentf-save.el
Wrote /home/user/.config/emacs/var/save-place.el
jotain.service: Main process exited, code=exited, status=255/EXCEPTION
```

### Root cause

`lisp/init-vc.el` added `jotain-git-stats--invalidate-current-buffer` to a
global `after-save-hook`, and the `jotain-git-stats` doom-modeline segment
calls `vc-git-root` on every redraw. Both call sites assumed `vc-git.el` was
loaded, but the `use-package vc-git` block sets only `:ensure nil` + `:bind`,
so the library autoloads on `C-x C-g`, not on init.

During first-run startup, `use-package :ensure t` calls `package-install`,
which writes `custom-file` via `package--save-selected-packages`. That fires
the global `after-save-hook`, the hook calls `vc-git-root`, the symbol is
void, and the daemon aborts before reaching `server-start`.

The two `declare-function` lines for `vc-git-root` in the file silenced the
byte-compiler but never arranged a runtime load. `vc-git-root` has no
`;;;###autoload` cookie upstream, so nothing implicitly loads it.

### Fix

`lisp/init-vc.el` — replaced both `(vc-git-root FILE)` calls outside the
interactive `jotain-switch-git-status-buffer` command with
`(locate-dominating-file FILE ".git")`. `locate-dominating-file` is in
`files.el` (always loaded) and is what `vc-git-root` calls through
`vc-find-root` internally; the only thing dropped is a `vc-file-getprop`
cache, which is immaterial at modeline-render frequency.

Also removed the now-unused second `(declare-function vc-git-root …)` above
the `with-eval-after-load 'doom-modeline` form. The first
`declare-function` and the `(require 'vc-git)` inside
`jotain-switch-git-status-buffer` remain, since that command keeps using
`vc-git-root` and demands the library at call time.

### Out of scope

The TLS errors against `elpa.gnu.org` / `elpa.nongnu.org` /
`stable.melpa.org` visible in the same journal are caused by the host's
corporate proxy. Emacs recovers from them and they were not the cause of
the abort; they only inflate startup wall time by ~50 s. Silencing the
eager refresh (`jotain--package-refresh-once` in `init.el`) when offline
is a separate, lower-priority change.
## Fix devenv shell, pull Emacs 31 from emacs-overlay

### Summary

Devenv shell evaluation failed with `error: attribute 'emacs31' missing`
because the pinned nixpkgs only ships `emacs30`. Wired
`nix-community/emacs-overlay` into both `flake.nix` and `devenv.yaml` and
switched `emacs.nix` to consume `pkgs.emacs-git` / `pkgs.emacs-macport` from
the overlay instead of the non-existent `pkgs.emacs31{-macport}`.

### Changes

- **`flake.nix`** — added `emacs-overlay` input (`inputs.nixpkgs.follows =
  "nixpkgs"`) and inserted `emacs-overlay.overlays.default` before
  `self.overlays.default` in `pkgsFor`.
- **`devenv.yaml`** — added the same `emacs-overlay` input, pinned to the
  rev `flake.lock` resolved to so both locks agree.
- **`devenv.nix`** — accepted the new `inputs` module arg and applied
  `inputs.emacs-overlay` before the project overlay. Added
  `nix-community` to the `cachix.pull` list (that is where
  `emacs-git` binaries live).
- **`emacs.nix`** — `basePackage` now selects `pkgs.emacs-git` for every
  variant except `macport`, which uses `pkgs.emacs-macport`. Standalone
  invocation (`nix-build emacs.nix`) re-applies the overlay through a new
  `overlays = [ (import …) ]` arg in the default `pkgs` derivation,
  pulling the rev from `flake.lock`. Fixed `srcRepo` default to include
  `mainline` so autoreconf runs (emacs-overlay's `emacs-git` is a git
  checkout). Cache-parity comment block updated to reference
  `pkgs.emacs-git` instead of the no-longer-existing `pkgs.emacs31`.
- **`Justfile`** — added `emacs-overlay` to `shared_inputs` so
  `just update` and `just verify` keep both locks in sync on that rev.
- **`CLAUDE.md`**, **`docs/architecture/overview.mdx`**,
  **`docs/architecture/nix-build.mdx`**, **`docs/installation.mdx`**,
  **`docs/introduction.mdx`** — replaced stale `pkgs.emacs30` /
  `pkgs.emacs30-macport` references and re-targeted the cache-parity
  invariant onto `pkgs.emacs-git`.

### Verification

- `nix-instantiate --eval` proves `(import ./emacs.nix {}).outPath ==
  pkgs.emacs-git.outPath` for the pinned overlay rev — the mainline
  build remains a binary-cache hit on `nix-community.cachix.org`.
- `devenv shell -- emacs --batch --version` reports `GNU Emacs 32.0.50`
  (the master branch has already moved past 31; `emacs-git` advertises
  itself externally as `20260519.0`).
- All seven assertions from `devenv.nix`'s `enterTest` block pass when
  run manually inside the shell (`emacs`, `emacsclient`, `etags` resolve
  into the `jotain-emacs-full` store path; `--init-directory` loads this
  repo's config; `eask` is on PATH).
- `nix flake check --no-build` evaluates clean.
- `just verify` reports OK on every shared input (`nixpkgs`,
  `treefmt-nix`, `emacs-overlay`).

### Notes

- The user-level `~/.config/nix/nix.conf` does not list
  `nix-community.cachix.org` as a substituter, so the very first
  `devenv shell` after this change still has to fetch the
  `emacs-git` binary via the explicit `cachix.pull` declaration (devenv
  layers it in). Users without devenv can either add the substituter to
  `nix.conf` directly or pass `--option extra-substituters
  https://nix-community.cachix.org --option extra-trusted-public-keys
  nix-community.cachix.org-1:mB9FSh9qf2dCimDSUo8Zy7bkq5CX+/rkCWyvRCYg3Fs=`.
- `emacs-overlay`'s `emacs-git` tracks master and is currently
  internally `32.0.50`. The repo's "Emacs 31+" framing still holds since
  `emacs ≥ 31` is satisfied; bumping to a tagged `emacs-31` would
  require waiting for the upstream release.
