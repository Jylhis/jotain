# 2026-06-09

## Fix `jotain.service` daemon crash (exit 255) + make package fetching optional

### Summary

The `jotain.service` systemd user unit on an offline host (`GEOZRHLNXJYL02`)
was in a restart loop, exiting `255/EXCEPTION` on every start. The journal
showed MELPA/NonGNU archive downloads failing, which *looked* like the cause,
but those failures are caught and non-fatal. The real fatal error was
`tagref-mode`.

### Root cause

`init-prog.el` hooked `tagref-mode` onto **every** `prog-mode` buffer
unconditionally. `tagref.el` (`tagref-mode`) calls
`(user-error "Not in a project; tagref-mode requires project.el")` when there is
no project. During daemon startup `*scratch*` is created in
`lisp-interaction-mode` (a `prog-mode` descendant); with the unit's
`default-directory` = `$HOME` (not a project) the hook fired and signalled. This
happens in `command-line` **before** `--fg-daemon`'s `server-start`/sd_notify,
so the daemon died un-notified → systemd reported `255/EXCEPTION`.

Confirmed by the journal ordering: `"Not in a project; tagref-mode requires
project.el"` → `"Wrote …/var/recentf-save.el"` (only runs from
`kill-emacs-hook`) → exit 255. `init.el` had fully loaded, so the melpa download
failures never aborted startup. The bug stayed hidden on dev machines because
the daemon there usually starts inside the repo (a project), so `tagref-mode`
enables cleanly.

### Changes

- **`lisp/init-prog.el`** — `tagref` `use-package` block: replaced
  `:hook (prog-mode . tagref-mode)` with `:hook (prog-mode .
  jotain-tagref--maybe-enable)`, a guard (defined in `:init`, with
  `:commands (tagref-mode)` for the autoload) that enables `tagref-mode` only
  when `(project-current)` is non-nil. Outside a project it declines silently
  instead of signalling. This is the crash fix.
- **`init.el`** — reworked archive fetching to be optional, background, and
  non-fatal (per request):
  - New `jotain-refresh-package-archives` defvar, defaulting off when the
    `JOTAIN_NO_PACKAGE_REFRESH` env var is set — the per-host opt-out for
    offline machines, no shared-config edit needed.
  - On-demand refresh advice now removes itself *before* refreshing and wraps
    `package-refresh-contents` in `with-demoted-errors`, so a dead network is a
    log line, not a fatal error, and can't re-trigger on every later install.
  - Added a background warm-up on `emacs-startup-hook` via `run-with-idle-timer`
    using async `(package-refresh-contents t)`, also demoted — never blocks the
    daemon, never fails the service.

### Follow-up for the offline host

Optional: set `Environment=JOTAIN_NO_PACKAGE_REFRESH=1` on the `jotain.service`
unit (or via `services.jotain`) to silence the now-harmless download-failure log
lines. Not changed in the shared default.

## Silence the corfu/ispell `completion-at-point` error storm

### Summary

The Messages buffer was flooded with repeating `Corfu detected an error: …
ispell-lookup-words: No plain word-list found at system default locations`
backtraces — one per keystroke in any text/YAML/Markdown/`COMMIT_EDITMSG`
buffer. Corfu catches the error, so it's non-fatal, but it spams the log and
silently disables in-buffer completion for those buffers.

### Root cause

Emacs 30 added the `text-mode-ispell-word-completion` option, which **defaults
to `completion-at-point`**. With that default, `text-mode` adds
`ispell-completion-at-point` to `completion-at-point-functions` in every
text-mode-derived buffer. `ispell-completion-at-point` → `ispell-lookup-words`
reads `ispell-alternate-dictionary`, whose default is `/usr/share/dict/words` —
**absent on NixOS**. So every corfu auto-completion in a prose buffer throws.
We spell-check with jinx, never with the ispell capf, so the feature is pure
breakage here.

### Changes

- **`lisp/init-writing.el`** — added `(text-mode-ispell-word-completion nil)` to
  the `text-mode` `use-package` `:custom` block, with a comment explaining the
  NixOS word-list gap. Verified: with the option nil, `ispell-completion-at-point`
  is no longer present in a text-mode buffer's `completion-at-point-functions`.

### Not changed (separate, deployment-dependent)

- **`Jinx: No dictionaries available for "en_US"`** — enchant has no en_US
  dictionary backend on this host (no `enchant`/dict on PATH). Fix belongs at
  the deployment layer (provide a hunspell/nuspell en_US dict + `DICPATH`, or
  switch the daily Emacs to the Nix-wrapped `jotainEmacsPackages` and add the
  dict there). Left for a follow-up once the daily-Emacs deployment path is
  confirmed.
- **`Failed to download 'melpa'/'melpa-stable' archive`** — the same transient,
  non-fatal background archive warm covered above; not a regression.

## Drop `withJansson` from `emacs.nix` (nixpkgs `make-emacs.nix` removed it)

### Summary

`nix flake check --all-systems` failed during package evaluation with
`function 'anonymous lambda' called with unexpected argument 'withJansson'`
pointing at nixpkgs' `make-emacs.nix`. Surfaced after the in-progress flake
update (nixpkgs → `8c3cede7`, emacs-overlay → `2aba9beb`).

### Root cause

Upstream nixpkgs removed the `withJansson` argument from
`pkgs/applications/editors/emacs/make-emacs.nix` — Emacs 30+ has built-in
JSON, so the `--with-jansson` flag is gone. `emacs.nix` still declared
`withJansson ? false` and forwarded it through `basePackage.override`, so the
override threw on the now-unknown argument. Confirmed `withJansson` is absent
from the pinned `make-emacs.nix` formal argument list; every other flag we
forward is still accepted.

### Changes

- **`emacs.nix`** — removed the `withJansson ? false` parameter and its entry
  in the `basePackage.override` `inherit` block. No other forwarded flag
  changed.

### Verification

- Cache-parity invariant still holds: `(import ./emacs.nix {}).outPath ==
  pkgs.emacs-git.outPath` evaluates to `true`, so the mainline build remains a
  binary-cache hit.
- Committed alongside the nixpkgs/emacs-overlay bump that triggered it;
  `just verify` confirms `flake.lock` and `devenv.lock` agree on all shared
  inputs.
