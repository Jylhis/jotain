## Fix `eglot-tempel-mode` void-function error when opening a `.nix` file

### Symptom

Opening a `.nix` file (e.g. `package.nix`) fires `nix-ts-mode` â†’
`eglot-ensure`, which loads eglot, and produced:

```
Error (use-package): eglot-tempel/:init: Symbol's function definition is void: eglot-tempel-mode
```

The eglot-tempel integration (Tempel-based expansion of LSP-sent snippets)
never armed and every first eglot session printed the error.

### Root cause

The `eglot-tempel` block in `init-snippets.el` registered two competing
`with-eval-after-load 'eglot` blocks:

```elisp
(use-package eglot-tempel
  :after eglot
  :functions (eglot-tempel-mode)
  :init (with-eval-after-load 'eglot (eglot-tempel-mode 1)))
```

`:init` runs unconditionally when `init-snippets.el` loads, registering a
block that calls `(eglot-tempel-mode 1)`. Separately, `:after eglot` (which
implies deferred loading) registers a *second* block that `require`s the
`eglot-tempel` package, the thing that actually defines `eglot-tempel-mode`.
`eval-after-load` runs blocks in registration order, and `:init` is expanded
before the `:after` load logic, so the `eglot-tempel-mode` call fired *before*
the package was required â†’ void function. The autoload did not cover it.

### Fix

Enable the mode from `:config` instead. Under `:after eglot`, use-package
already requires `eglot-tempel` when eglot loads, and `:config` runs after
that require, so the function is defined by the time it is called. The inner
`with-eval-after-load 'eglot` was redundant and is gone:

```elisp
(use-package eglot-tempel
  :after eglot
  :functions (eglot-tempel-mode)
  :config (eglot-tempel-mode 1))
```

`:functions (eglot-tempel-mode)` stays â€” the repo byte-compiles with
`byte-compile-error-on-warn` and without packages on `load-path`, so the
`declare-function` it emits keeps the `:config` body from erroring as an
unknown function. The mode still arms while eglot is being required (before
it connects), so the "armed the moment eglot loads" comment remains accurate.
The separate `jotain-tempel-eglot-capf` merge in the `tempel` block is
untouched: it merges the user's own snippet capf into corfu, which is
complementary to `eglot-tempel-mode`'s server-snippet expansion.

## Auto-adopt any language a project's devenv provides (eglot + envrc)

### Motivation

In a project whose `devenv.nix` enables `languages.go.enable = true` (or
rust/python/haskell/c/â€¦), devenv puts the toolchain and its LSP server on
`PATH`. `envrc-global-mode` already applies that per-project `PATH`
buffer-locally, but eglot only auto-started for a fixed hook list
(dockerfile, go, nix, python, rust, ts/tsx, zig). Languages with a mode but
no hook (C/C++, Haskell) or no mode wired at all never lit up, and the
`rass`/basedpyright/typescript gating was decided once at startup against
Jotain's *own* shell â€” so a daemon started outside devenv never registered it.

### Change (`init-prog.el`, eglot block)

1. **General devenv-aware auto-start.** New `jotain-prog--maybe-eglot-ensure`
   on `prog-mode-hook` (in `:init`, outside eglot's deferral) probes, on an
   idle timer so envrc's env is applied first, whether
   `eglot--guess-contact` resolves a server whose binary is on the buffer's
   `exec-path`; if so it calls `eglot-ensure`. Uses eglot's built-in
   `eglot-server-programs` table (clangd, gopls, rust-analyzer, HLS, zls,
   bash-language-server, solargraph, â€¦). Skips remote/non-file/Lisp buffers
   and already-managed buffers, so no eglot load or warning spam in elisp.

2. **Connect-time server discovery.** The startup-time `(when (executable-find
   "rass") â€¦)` block is replaced by function-valued `eglot-server-programs`
   contacts (`jotain-prog--ts-server`, `jotain-prog--python-server`). eglot
   evaluates them at connect time in the buffer, so `rass`/basedpyright/ruff
   discovery honours the project's devenv `PATH`, not Jotain's. Each falls
   back to a plain single server when the multiplexer is absent. The
   `jotain-prog-enable-risky-js-lsp` opt-in is preserved via
   `jotain-prog--risky-js-extras`.

The explicit curated `:hook` list stays as a zero-regression safety net;
`eglot-ensure` is idempotent so it composes with the general hook.

### envrc ergonomics (`init-prog.el`, envrc block)

Bound `C-c e` â†’ `envrc-command-map` (envrc's own recommendation): `C-c e a`
allow, `C-c e r` reload, `C-c e R` reload-all (picks up a terminal `direnv
allow` in a running Emacs). Replaces reaching for `M-x envrc-allow` per
project. Manual allow stays the secure default â€” no auto-allow whitelist.

### Prerequisite

The target project must expose devenv's env to Emacs: an `.envrc` containing
`use devenv` (+ `direnv allow`), or Emacs launched from inside `devenv shell`.
Without one, nothing is on `PATH` and no mechanism can help.

### Verification

`byte-compile-error-on-warn` full compile of `early-init.el init.el
lisp/init-*.el` exits clean. (The Nix `elisp-compile` flake check is
temporarily blocked by the untracked in-progress `lisp/init-lang-go.el`, which
Nix excludes from the git source â€” unrelated to this change.)

## Overhaul Go development experience

### Motivation

Go was the least-developed language in the config and inconsistent with the
Rust/Python bar. Concrete gaps:

- `init-lang-systems.el` pulled the third-party `go-mode` from MELPA even
  though the file's own comment claimed Go was "built in (ensure nil)".
  `treesit-auto` + built-in `go-ts-mode` already owned `.go`, so `go-mode`
  was redundant.
- Inlay hints were a no-op: `eglot-inlay-hints-mode` was enabled for Go, but
  gopls emits nothing without a `hints` workspace config, and there was no
  gopls `eglot-workspace-configuration` at all.
- `compile-multi` keyed its Go commands on `go-mode`, but buffers open in
  `go-ts-mode` (Emacs 30 does not make `go-ts-mode` a child of `go-mode`),
  so the picker never fired.
- No formatter wiring, no debugging, no Go helpers.

### Changes

- **New `lisp/init-lang-go.el`** (Go graduated to its own file like
  nix/rust/python). Registers built-in `go-ts-mode` + `go-mod-ts-mode` from a
  single `use-package` block (`go-mod-ts-mode` lives inside `go-ts-mode.el`,
  so a separate `:ensure nil` block would try to load a non-existent feature â€”
  a mistake caught by the byte-compile log). No indent-offset is set: gofmt
  uses tabs and the built-in default matches. Adds `go-tag` and `gotest`
  under a buffer-local `C-c C-t` Go-tooling prefix (chosen so it never
  shadows the global `C-c t` theme toggle).
- **`init-prog.el`**: eglot hook + inlay-hints list + `gopls` server override
  re-keyed to `go-ts-mode`/`go-mod-ts-mode` (dropped `go-mode`). Added a
  `:gopls` `eglot-workspace-configuration` (usePlaceholders, completeUnimported,
  staticcheck, the full `hints` set, and unusedparams/shadow/nilness/unusedwrite
  analyses) â€” this is what actually lights up inlay hints and staticcheck via
  flymake. apheleia now maps `go-ts-mode â†’ gofmt` explicitly. Added `dape`
  (Debug Adapter Protocol) under a new `C-c d` prefix; its built-in Delve
  config drives Go debugging via `dlv dap`.
- **`init-project.el`**: `compile-multi` Go entry re-keyed to `go-ts-mode` and
  enriched (`go test ./...`, current, `-race`, `go vet`, `go build ./...`).
- **`init-lang-systems.el`**: removed the `go-mode` block; header updated.

### Keybinding reshuffle (docker removed to free `C-c d`)

`C-c d` was triple-claimed: `docker` (devops), `dirvish` (navigation), and a
which-key label of "dirvish" â€” with devops loading last, docker silently won.
The docker Magit-style TUI was removed (its `jotain-docker-backend` defcustom
stays, trimmed to only drive `dockerfile-mode`). `C-c d` is now the `dape`
debug prefix; `dirvish` moved to `C-c f` (`dirvish-side` stays on `C-c D`).
`C-c e` was the obvious target but is already the envrc command-map prefix, so
`C-c f` ("files") was used instead. which-key labels and
`docs/keybindings.mdx` updated to match.

### External binaries (from the project env, not bundled in Nix)

`gopls` (LSP), `dlv` (dape), `gotests` (gotest generate), `gomodifytags`
(go-tag) â€” same "LSP from project environment" philosophy as Rust/Python.
`gofmt` ships with the Go toolchain.

### Verification

`nix build .#checks.x86_64-linux.elisp-compile` passes (needs
`lisp/init-lang-go.el` git-staged so the flake source sees it). `go-tag`,
`gotest`, and `dape` all resolve in the pinned `emacsPackages`, so the
use-package Nix auto-mapping pulls them in with no `trivialBuild` entry.

## Log when a classic (non-tree-sitter) major mode is used despite a ready ts-mode

### Motivation

Files are routed to `*-ts-mode` unevenly: `treesit-auto` +
per-language `:mode` entries cover the well-developed languages, but
several classic modes (C/C++, classic `yaml-mode`, `web-mode`,
`terraform-mode`, â€¦) still win even though a built-in `*-ts-mode` and its
grammar (Nix ships them via `$TREE_SITTER_DIR`) are available. Nothing
surfaced these gaps, so they were invisible.

### Change (`init-prog.el`, new "Non-tree-sitter mode diagnostic" subsection)

A `jotain-prog--warn-non-ts-mode` function on `after-change-major-mode-hook`
logs to `*Messages*` whenever the buffer's `major-mode` is a classic mode
that has a tree-sitter counterpart whose grammar is actually loadable, e.g.:

```
jotain: foo.sh opened in sh-mode; bash-ts-mode (tree-sitter, grammar `bash') is available
```

Design notes:

- **Reuses `treesit-auto-recipe-list`** (already loaded `:demand t`) as the
  authoritative classicâ†”tsâ†”grammar source rather than hardcoding a table.
  A hash table (classic-mode â†’ `(ts-mode . lang)`) is built **once** at load,
  so the hook is an O(1) `gethash` on the common miss path â€” matching the
  repo's aversion to per-find-file overhead (the reason
  `global-treesit-auto-mode` was rejected). `:remap` may be a bare symbol
  (`sh-mode`) or a list (`(js-mode javascript-mode js2-mode)`), normalised
  with `proper-list-p`.
- **`treesit-ready-p` guard is required, not optional.** Every `*-ts-mode`
  symbol is always `fboundp` even with no grammar installed, so `fboundp`
  alone would nag for every classic mode on any grammar-less host;
  `treesit-ready-p` returns nil when the grammar is missing, so the notice
  only fires when tree-sitter would genuinely work. `major-mode-remap-defaults`
  was rejected as a source â€” with grammars absent it self-populated only
  C/C++ entries; Python/JS/CSS/YAML never register there.
- Always on via `defcustom jotain-prog-warn-non-ts-mode` (default `t`);
  `setopt` it to nil to silence. Byte-compile-clean under
  `byte-compile-error-on-warn` via `(eval-when-compile (require 'treesit-auto))`
  (cf. the `doom-modeline` block in `init-vc.el`) plus `declare-function`
  for the struct accessors and `treesit-ready-p`.

### Verification

Full `byte-compile-error-on-warn` compile of `early-init.el init.el
lisp/init-*.el` exits clean. Runtime probe against `jotainEmacsPackages`:
the table builds with 64 entries and the irregular/list cases resolve
correctly (`sh-mode â†’ (bash-ts-mode . bash)`, `js-mode`/`javascript-mode â†’
(js-ts-mode . javascript)`), while `python-ts-mode` is absent as a key (so
already-ts buffers never nag) and the hook is installed. The Nix
`nix build .#checks.x86_64-linux.elisp-compile` flake check passes.
