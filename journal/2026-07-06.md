# 2026-07-06

## Fix `eglot-tempel-mode` void-function error when opening a `.nix` file

### Symptom

Opening a `.nix` file (e.g. `package.nix`) fires `nix-ts-mode` →
`eglot-ensure`, which loads eglot, and produced:

```
Error (use-package): eglot-tempel/:init: Symbol's function definition is void: eglot-tempel-mode
```

The eglot-tempel integration (Tempel-based expansion of LSP-sent snippets)
never armed and every first eglot session printed the error.

### Root cause

The `eglot-tempel` block in `init-snippets.el` registered two competing
`with-eval-after-load 'eglot` blocks:

```elisp
(use-package eglot-tempel
  :after eglot
  :functions (eglot-tempel-mode)
  :init (with-eval-after-load 'eglot (eglot-tempel-mode 1)))
```

`:init` runs unconditionally when `init-snippets.el` loads, registering a
block that calls `(eglot-tempel-mode 1)`. Separately, `:after eglot` (which
implies deferred loading) registers a *second* block that `require`s the
`eglot-tempel` package, the thing that actually defines `eglot-tempel-mode`.
`eval-after-load` runs blocks in registration order, and `:init` is expanded
before the `:after` load logic, so the `eglot-tempel-mode` call fired *before*
the package was required → void function. The autoload did not cover it.

### Fix

Enable the mode from `:config` instead. Under `:after eglot`, use-package
already requires `eglot-tempel` when eglot loads, and `:config` runs after
that require, so the function is defined by the time it is called. The inner
`with-eval-after-load 'eglot` was redundant and is gone:

```elisp
(use-package eglot-tempel
  :after eglot
  :functions (eglot-tempel-mode)
  :config (eglot-tempel-mode 1))
```

`:functions (eglot-tempel-mode)` stays — the repo byte-compiles with
`byte-compile-error-on-warn` and without packages on `load-path`, so the
`declare-function` it emits keeps the `:config` body from erroring as an
unknown function. The mode still arms while eglot is being required (before
it connects), so the "armed the moment eglot loads" comment remains accurate.
The separate `jotain-tempel-eglot-capf` merge in the `tempel` block is
untouched: it merges the user's own snippet capf into corfu, which is
complementary to `eglot-tempel-mode`'s server-snippet expansion.
