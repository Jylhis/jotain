---
title: Modules Overview
description: Understanding Jotain's modular architecture
---

Jotain's Emacs configuration is split into functional modules, each covering a specific domain. Modules live in `elisp/` and are loaded by `init.el` via `require` in a defined order.

## Module load order

Modules are loaded sequentially — later modules can depend on earlier ones:

| Module | Purpose |
|--------|---------|
| `platform` | Platform detection helpers (loaded first) |
| `core` | Base settings, keybindings, built-in modes |
| `fonts` | Font selection and rendering |
| `ui` | Theme system, dashboard, frame setup |
| `dashboard` | Startup screen (Enlight) |
| `completion` | Vertico, Corfu, Consult, Orderless |
| `programming` | Eglot (LSP), tree-sitter, DAP, language modes |
| `ai` | AI coding assistants |
| `per-project` | Project-local configuration |
| `writing` | Org-mode, documentation tools |
| `git` | Magit integration |
| `help` | Enhanced help system |
| `systems` | System administration tools |
| `platforms` | Linux/macOS/Android adaptations |

<Note>
  `utils.el` is not loaded directly by `init.el` — it's required by `writing.el`. `android.el` is loaded conditionally on Android platforms.
</Note>

## Module conventions

Every module follows these conventions:

1. **Provide**: Each module ends with `(provide 'module-name)`
2. **Require**: Loaded via `(require 'module-name)` in `init.el`
3. **Load order matters**: Modules can depend on earlier modules in the load sequence
4. **use-package**: All package configuration uses `use-package` forms
5. **Nix-only packages**: Use `:ensure nil` for built-in packages, never `:ensure t` (packages are provided by Nix)
